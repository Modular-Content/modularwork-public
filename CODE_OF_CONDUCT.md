# 1. Purpose

ModularWork is built around collaboration, engineering discipline and long-term development.

This document defines the expected behavior of all participants interacting with the ModularWork ecosystem.

The goal is simple:

> Build technology together without unnecessary conflicts, hostility or disruption.

---

# 2. Core Principles

## 2.1 Respect

All participants MUST:

- communicate professionally;
- respect different levels of experience;
- criticize implementations, not people;
- provide constructive feedback.

Technical disagreement is allowed.

Personal attacks are not.

---

## 2.2 Engineering Over Ego

Decisions are based on:

```
Evidence
   >
Reasoning
   >
Experience
   >
Personal Preference
```

Arguments such as:

- "I like it this way"
- "everyone does it differently"
- "this is how I always code"

are not sufficient technical justification.

---

## 2.3 Quality Over Speed

The project prioritizes:

```
Correctness
      >
Maintainability
      >
Consistency
      >
Convenience
```

Rushed changes that damage architecture are discouraged.

---

# 3. Unacceptable Behavior

The following actions are prohibited:

## Communication

❌ Personal attacks  
❌ Harassment  
❌ Insults  
❌ Provoking conflicts intentionally  
❌ Discrimination  

---

## Development

❌ Intentionally breaking systems  
❌ Hiding bugs or vulnerabilities  
❌ Sabotaging contributions  
❌ Ignoring documented project rules  

---

## Community

❌ Spam  
❌ Excessive self-promotion  
❌ Manipulation of discussions  
❌ Creating unnecessary drama  

---

# 4. Technical Discussions

Disagreements are expected in software development.

A disagreement MUST focus on:

```
Problem
   |
   v
Constraints
   |
   v
Possible Solutions
   |
   v
Decision
```

Not:

```
Person
   |
   v
Attack
```

---

# 5. Maintainer Responsibility

Maintainers SHOULD:

- explain important decisions;
- enforce project standards consistently;
- avoid personal bias;
- treat contributors fairly.

Maintainers MAY reject contributions that:

- violate architecture;
- reduce maintainability;
- ignore project standards.

---

# 6. Enforcement

Violations may result in:

| Level | Action |
|-|-|
| Minor | Warning |
| Repeated | Temporary restriction |
| Severe | Removal from project spaces |

Actions depend on:

- severity;
- intent;
- previous behavior.

---

# 7. Reporting Problems

If a participant experiences unacceptable behavior:

1. Do not escalate publicly.
2. Contact project maintainers.
3. Provide relevant context.
4. Allow maintainers to investigate.

---
